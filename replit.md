# Overview

This project is a pnpm workspace monorepo using TypeScript, designed to build a professional-grade crypto exchange platform primarily targeting the Indian market. It includes a mobile application, a comprehensive admin web panel, and a desktop web user portal. The platform aims to offer spot and future trading, multi-wallet support, KYC verification (L0-3), VIP tiers, Earn programs, and facilitate INR and crypto deposits/withdrawals, adhering to Indian tax regulations (1% TDS).

The business vision is to capture a significant share of the Indian crypto exchange market by providing a feature-rich, user-friendly, and compliant platform. The project ambitions include high scalability, robust security, and a seamless user experience across all interfaces.

# System Architecture

The project is structured as a pnpm workspace monorepo, leveraging Node.js 24 and TypeScript 5.9.

**UI/UX Decisions:**
- **Mobile App:** Designed for a Binance/CoinDCX style experience with a dynamic theme system and consistent branding.
- **Admin Panel:** Premium Zebvix Admin Console with a dark navy + gold/amber Binance-style theme across 28 pages, featuring a branded sidebar, sticky header with command palette, and role-gated access. Includes a premium dashboard with KPI cards, pending approvals breakdown, system health monitoring, and user/withdrawal teasers. Reusable premium components and custom CSS utilities are implemented.
- **User Portal:** Utilizes a dark gold/amber theme (Binance-style) for Home, Markets, Trade, and Futures terminals with `lightweight-charts` for data visualization and live price feeds.
  - **Account Pages:** Comprehensive profile management (KYC, bank accounts, settings with 2FA, sessions), Earn programs with KYC gating, and an Invite & Earn system with referral tracking and commission tiers. All pages use `react-query` and a consistent amber/orange brand gradient.
  - **Production Hardening:** Includes stub pages for future features (`/p2p`, `/convert`), rewritten Orders and Portfolio sections using shared premium primitives, and a robust `ErrorBoundary` for improved user experience.
  - **Order Contracts:** Standardized API endpoints for Spot (`/exchange/order`) and Futures (`/futures/order`, `/futures/position`) trading, with wallet balance (`/finance/wallet`) available for derived `available` balances.
  - **Shared Auth:** `bicryptoAuth` middleware accepts both `cx_session` cookie and Bearer JWT for unified authentication across web and Flutter clients.
  - **WS Trades Stream:** Symbol-scoped `trades:<symbol>` WebSocket stream for real-time market data.

**Technical Implementations:**
- **API Framework:** Express 5 with cookie sessions, `bcryptjs` for authentication, and role-based access control.
- **Database:** PostgreSQL with Drizzle ORM, featuring a comprehensive schema across 12 modules.
- **Validation:** Zod (`zod/v4`) with `drizzle-zod` for robust data validation.
- **API Codegen:** Orval generates API hooks and Zod schemas from an OpenAPI specification.
- **Build System:** esbuild for CJS bundle generation.
- **Authentication:** HMAC-SHA256 JWTs with platform-specific token persistence (SharedPreferences for web, `flutter_secure_storage` for native). OTP codes are hashed and generated cryptographically.
- **Real-time Data:** Custom `marketSocket.ts` normalizes Bicrypto-style WebSocket frames.
- **Monetary Transactions:** Atomic database transactions with `SELECT FOR UPDATE` and `onConflictDoUpdate` for data integrity in all sensitive financial operations.
- **Security Hardening:** OTP codes are cryptographically generated, hashed at rest, and consumed atomically. KYC approval is monotonic to prevent level downgrades.
- **Request Correlation:** `X-Request-Id` header (or UUID fallback) is used for request tracing and logging across services.
- **Liveness vs Readiness Probes:** Separate endpoints (`/api/healthz` and `/api/readyz`) for cheap liveness checks and deep dependency checks (Postgres, Redis).
- **Zod Request Validation:** Middleware for API request body validation with detailed error responses.
- **Admin Hardening:** Global `ErrorBoundary` for admin UI, automatic redirection to login on 401, and replacement of native dialogs with `shadcn` components.
- **Multi-server / Horizontal Scaling:**
    - **Leader Election:** Redis-based leader election using `SET key NX EX` with atomic heartbeats to ensure a single leader for critical tasks.
    - **Leader-gated Workers:** Recurring tasks (price-service, sweepers, bot-service, futures engine ticks) are executed only by the elected leader.
    - **WS Fanout:** Leader publishes price ticks to Redis pub/sub, and followers subscribe to fan out to their connected WebSocket clients.
    - **Distributed Rate Limits:** `express-rate-limit` configured with `rate-limit-redis` for shared, distributed rate limiting across all replicas.
    - **Bootstrap Order:** Strict initialization sequence ensuring Redis connection and leader election before app import and server start.
    - **Single-instance Fallback:** Graceful degradation to single-instance operation if Redis is unavailable.
    - **Environment Variables:** Configurable Redis settings for multi-server deployments.

**Feature Specifications:**
- **Auth System:** Comprehensive authentication with `/auth`, `/admin`, `/public` routers, supporting both mobile and web clients.
- **KYC System:** Multi-level verification with admin approval workflow.
- **Wallets and Banking:** Multi-wallet support, INR/crypto deposits/withdrawals, and single-verified-bank rule with IFSC validation.
- **Trading Features:** Spot and Futures trading terminals, order placement, and real-time market data.
- **Admin Fund Endpoint:** `adminOnly` API for atomically funding user wallets with audit trails.
- **Bicrypto v5 Adapter:** Provides Flutter-shaped endpoints for various functionalities, including auth with Proof-of-Work.
- **Go Service Skeleton:** Established for performance-critical services like the matching engine.
- **In-Memory Matching Engine:** A pure-TypeScript, single-threaded, low-latency matching engine lives at `artifacts/api-server/src/lib/inmem-engine/` (orderbook.ts / pricelevel.ts / engine.ts / wal.ts / snapshot.ts). Uses sorted price arrays + per-level doubly-linked FIFO queues for O(log n) cross + O(1) maker pop; supports limit / market / IOC / FOK / post_only orders, self-trade prevention (cancel_newest|cancel_oldest|cancel_both), partial fills, price-time priority, single-threaded event-queue processing, append-only JSONL WAL (with optional `fdatasync` per append) + atomic snapshot/recovery, and per-symbol halt/resume (WAL-logged). Benchmarked at ~232–263k orders/sec, p99 latency 11–12µs (fsync off; well under the 1ms target). Exposed via admin-only HTTP at `/api/admin/inmem-engine/{orders,orderbook/:symbol,metrics,snapshot}`. Runs in PARALLEL to the existing Redis+Postgres engine in `lib/matching-engine.ts` — no production wallet impact from this raw engine; the production wrapper below is the one that touches real wallets.
- **Production Engine (in-memory + DB-settled):** A production-grade orchestrator wraps the raw in-memory engine at `artifacts/api-server/src/lib/inmem-engine/prod/` and exposes admin routes under `/api/admin/inmem-engine-prod/*`. It runs alongside `lib/matching-engine.ts` (no cutover yet), and adds:
  - **Symbol Registry** (`prod/symbol-registry.ts`): in-memory cache of `pairsTable` refreshed every 30s; enforces tickSize, lot/qty precision, minQty, maxQty, minNotional; resolves base/quote coin ids and fee rate buffers.
  - **Risk Guard** (`prod/risk.ts`): per-user token bucket (default 20 ord/sec sustained, 50 burst) plus `maxOpenOrders` cap (default 200). Returns typed errors for rate / max-open violations.
  - **ProdEngine Orchestrator** (`prod/prod-engine.ts`): single-flight `start()` (listeners attached BEFORE recovery to avoid dropped trade events), atomic `placeOrder` flow = registry validate → risk check → halt check → INSERT order row + `balance↓ / locked↑` move under `SELECT FOR UPDATE` → submit to engine. Maintains an in-memory `dbOrderId → engineOrderId` map populated at placement AND rebuilt from the engine's recovered resting orders (via `engine.allRestingOrdersBySymbol()`) so cancels stay O(1) across restarts. Buy locks reserve `qty * limitPrice * (1 + takerFeeRate * 1.5)` so the placement-time lock covers the worst-case fee even when the order trades as taker.
  - **Async Settler** (`prod/settler.ts`): event-driven queue that drains engine `trade` / `autoCancel` events; per trade it runs ONE Postgres TX (`SELECT FOR UPDATE` on both wallets and orders), debits/credits taker+maker quote and base wallets using the SAME fee buffer used at placement time, applies VIP/admin fee overrides + GST + 1% sell-side TDS via `getSpotFeeRates()`, inserts symmetric `tradesTable` rows, updates `ordersTable.filledQty/avgPrice/status`, and bumps `pairs.lastPrice/volume24h`. Persists a `lastSettledTradeId` cursor (de-dupes recovery-replayed trades) and writes JSONL dead-letter records on failure (book has already moved on, so failures never block matching). Logs a warning if effective fees ever exceed the lock buffer (would mean the buffer needs widening at the registry level).
  - **Wallet Lock Model:** `balance` is the FREE balance and `locked` is the RESERVED slice (total funds = balance + locked). Lock = `balance↓ + locked↑`; release = `balance↑ + locked↓`. Mirrors the legacy `routes/orders.ts` and `lib/matching-engine.ts` accounting so the new path is interchangeable.
  - **Admin HTTP** (`routes/inmem-engine-prod.ts`): `POST /orders`, `DELETE /orders/:dbId?userId=`, `POST /halt`, `POST /resume`, `POST /cancel-all`, `GET /orderbook/:symbol`, `GET /metrics`, `POST /snapshot`, `POST /shutdown`. All routes require admin/superadmin.
  - **Admin "Trading Engine" Console** (`artifacts/admin/src/pages/trading-engine.tsx`, route `/trading-engine`, nav under "Markets & Trading", icon `Cpu`): live operator surface for the prod engine. Polls `/metrics` every 3s and `/orderbook/:symbol` every 2s; renders 6 KPI cards (uptime, total trades, symbols, engine queue, match latency, settler queue), an Async Settler health card with status pill (Healthy / Backlog growing / Failures detected) and dead-letter dump, a Risk Guard card showing live counters and the `ordersPerSecond / burstSize / maxOpenOrders` config, a Halted-Symbols quick-pick row, an order book panel with depth bars + spread, and a per-symbol controls card (Halt/Resume + Cancel-All with confirm dialog). Side panel exposes a Place-on-Behalf form (Buy/Sell × Limit/Market/IOC/FOK/Post-Only with auto-coercion when side flips to BUY since market-buy is unsupported) and Cancel-by-DB-Id. Page-level actions: Force Snapshot + Shutdown (both confirm). Network failures on the polling queries surface explicit error banners + Retry buttons so an operator never mistakes a fetch failure for a healthy idle engine. Route is wrapped in a `RoleGated` boundary that allows only admin/superadmin (the global admin gate also accepts `support`).
  - **Known Limits:** Market BUY orders are intentionally rejected at `placeOrder()` (no quote-budget mode yet — would otherwise lock zero quote at price=0 and credit base for free); use limit / IOC / FOK with an explicit price for taker buys. Resting orders are persisted via the engine WAL+snapshot, NOT re-inserted from the DB, so a wiped data dir leaves DB-open orders as "ghosts" that can only be cancel-refunded (the cancel fallback handles this safely).
- **Admin Control & Audit (Phase 2 Track A+B short):**
  - **Per-session order rate limit** (`app.ts` `orderLimiter`): 120 req/min, keyed by session-token tail (or IP fallback), mounted on `/api/exchange/order` and POST/non-GET `/api/orders` BEFORE the global limiter so a single user can't burst-DoS the matching engine. Bearer-only requests bypass cookie keying via the IP fallback.
  - **Status-active gate** (`middlewares/auth.ts` `blockIfNotActive`): both `requireAuth` and `requireRole` now reject any session whose user row has `status != 'active'` with HTTP 403 `Account suspended`. Means a frozen user is locked out platform-wide on the very next request even if their cookie/JWT is still cryptographically valid.
  - **User Freeze / Unfreeze** (`POST /api/admin/users/:id/freeze` and `/unfreeze` in `routes/admin.ts`): freeze sets `status='suspended'` AND deletes all that user's session rows in one shot, returning `sessionsRevoked` count; refuses self-freeze (400) to prevent operator lockout. Surfaced in the Users dossier sheet (`pages/users.tsx`) as red Lock / green Unlock buttons; freeze opens an AlertDialog with a required-but-optional reason textarea that flows into the audit payload.
  - **Force Cancel Order** (`POST /api/admin/orders/:id/cancel`): operator-only spot-order cancel via new `adminCancelSpotOrderById()` in `routes/orders.ts` — same row-locked refund logic as user-side `cancelSpotOrderById()` but without the `userId`/`isBot` constraints, so it can act on any user's or bot's open/partial order. Surfaced in `pages/orders.tsx` as a role-gated red Action column with reason-textarea AlertDialog; only renders for OPEN/PARTIAL.
  - **Audit Log infra** (`lib/audit.ts` `logAdminAction({req, action, entity, entityId, payload})`): fire-and-forget INSERT into `auditLogsTable` with try/catch so a logging failure can never break the business operation. Wired onto `PATCH /admin/users/:id` (`user.update`), `POST /admin/users/:id/disable-2fa` (`user.disable_2fa`), `POST /admin/users/:id/force-logout` (`user.force_logout`), freeze (`user.freeze`), unfreeze (`user.unfreeze`), and force-cancel (`order.force_cancel`). Bank/deposit/withdrawal/key-reveal mutations are NOT yet wired (tracked as follow-up).
  - **Audit Log Viewer** (`pages/audit-log.tsx`, route `/audit-log`, nav under "System" with `History` icon): paginated table (limit 500) with filters for actor email, entity (user/order/wallet/etc), action, and entity-id; KPI cards for total / 24h / distinct actors / distinct entities (powered by `GET /api/admin/audit-logs/stats`); each row expands a `<details>` block showing the JSON payload pretty-printed. Server-side `GET /api/admin/audit-logs` hydrates `actor` (id+email+name) via a single `IN (...)` lookup so the UI doesn't N+1.
  - **Source / Schema Download** (`routes/admin-source.ts`): two new GET endpoints power the Code Reference page's download buttons — `GET /api/admin/source/download?root=<key>` streams a ZIP of any allow-listed root (admin / user-portal / api-server / lib/db) using `archiver` with the same `SKIP_DIRS` exclusions as the source-tree browser (no `node_modules`, `dist`, `.git`, etc.); `GET /api/admin/source/db/download` returns a single `.sql` file containing CREATE TABLE + indexes for every public table (schema only, NO data) generated via the same introspection helpers as the table-detail view. Both endpoints are admin/superadmin only and write a `source.download` row to the audit log so every dump is traceable. The Code Reference page (`pages/code-reference.tsx`) renders a "Download <root> (.zip)" or "Download schema (.sql)" button in the page header that switches based on the active tab — implemented as plain `<a download href>` so the browser uses the existing session cookie. While editing this file the pre-existing `subtitle` → `description` PageHeader prop typo was also fixed.
- **User-portal "More" Mega-Menu:** Header dropdown groups three discovery sections — Tools (Calculator, Currency Converter, Crypto Compare, Price Predictions), Promotion (Announcements), and Explore (Trading Leagues). Same grouping rendered in the mobile drawer. All six pages are public and use the premium pattern (`PageHeader`, `SectionCard`, `PremiumStatCard`, `StatusPill`); Predictions clamps projected prices to a sane band and explicitly carries a "NOT financial advice" disclaimer.
- **Options Trading (Black-Scholes derivatives):** Full-featured options product across 4 new tables — `option_contracts` (call/put, strike, expiry, IV bps, risk-free rate bps, contract size, status active/expired/settled), `option_orders`, `option_positions` (unique on userId+contractId+side+status='open'), `option_settlements`. Pricing engine `api-server/src/lib/options-pricing.ts` implements full Black-Scholes for calls and puts plus all greeks (delta, gamma, theta-per-day, vega-per-vol-point, rho-per-rate-point) with intrinsic-only fallback when ttYears ≤ 0 and IV floor at 0.0001 to avoid divide-by-zero. Settlement engine `api-server/src/lib/options-engine.ts` runs leader-gated every 60s: marks expired contracts, then for each one wraps payoff credit + position close + contract `status='settled'` in a single Postgres TX (so a crash mid-loop rolls back, preventing double-credit). User routes at `/api/options/*` (contracts/quote/orders/positions/close/history) — order placement uses `db.transaction` with `SELECT FOR UPDATE` on the USDT spot wallet to prevent double-spend; buy debits the premium, sell-to-open locks `max(spot, strike)*qty` USDT margin. Admin routes at `/api/admin/options/*` (CRUD + force-settle + IV inline edit). User UI at `/options` (chain table with calls/puts side-by-side, expiry picker, order ticket modal with Hinglish copy, live positions table with mark+greeks+close, history). Admin UI at `/options-admin` (KPI cards, contract list, create-dialog, IV inline edit, force-settle, delete).
- **Web3 Multi-Chain Trading (8 chains, simulated on-ledger):** New tables `web3_networks` (8 seeded — eth, bsc, polygon, arbitrum, optimism, base, avalanche, solana), `web3_tokens` (33 seeded — natives + USDT/USDC + 2-3 majors per chain, each pinned to a `priceCoinSymbol` from the existing `coins` table), `web3_wallets` (watch-only addresses with family-aware base58/hex regex validation), `web3_swaps`, `web3_bridges`. Quote engine `api-server/src/lib/web3-quote.ts` reads live prices from `coinsTable` to compute swap rates with slippage caps + LP fee bps + per-network gas USD; bridge quote applies a flat bridgeFee bps + sum of source+dest gas. **Important: this is a synthetic exchange-side product, NOT real on-chain swaps** — the swap moves balances between the user's existing spot wallets keyed on the token's `priceCoinSymbol` (so swapping 500 USDT→ETH debits the spot USDT wallet by 500 and credits the spot ETH wallet by ~0.213); a fake `0x…` or base58 txHash is generated for UX realism. All wallet movements use `db.transaction` + `SELECT FOR UPDATE` and create the destination spot wallet on-the-fly if missing. Routes at `/api/web3/*` (networks/tokens/wallets/quote/swap/bridge/quote/bridge/swaps/bridges/portfolio). Admin routes at `/api/admin/web3/*` (network CRUD + token CRUD + status/fee inline edit). User UI at `/web3` (Tabs: Swap | Bridge | Wallets | History — premium chain pickers, slippage controls, route hints like "Uniswap V3 / 1inch route" or "LayerZero / Stargate"). Admin UI at `/web3-admin` (Networks tab + Tokens tab with network filter, status toggle, fee inline edit).
- **Express `req.user` typing fix:** `middlewares/auth.ts` previously declared `declare module "express-serve-static-core"` which silently failed at typecheck (TS2664: `Invalid module name in augmentation`) because `@types/express-serve-static-core` is not a direct dep of api-server. Replaced with the standard `declare global { namespace Express { interface Request { user?: User } } }` pattern, which fixed ~25 pre-existing `req.user` typecheck errors across security.ts / transfer.ts / public.ts / public-user.ts / support-user.ts / redis-admin.ts / v1.ts / auth.ts / api-key-auth.ts and unblocks correct typing for the new options.ts and web3.ts routes.
- **DexScreener-style Auto-Listing System:** Automated token discovery + listing pipeline driven by admin-configurable rules. Three new tables (`lib/db/src/schema/listings.ts`): `listing_sources` (CoinGecko top markets + DexScreener boosts/latest, with per-source `intervalSec` + `maxItemsPerSync` + `nextRunAt` cursor), `listing_rules` (per-rule `mode=auto|manual|off`, `scope=spot|web3|both`, min volume/mcap/liquidity/age thresholds, `chainsAllowed`/`sourceFilter` whitelists, `autoCreatePair` + `quoteSymbol` for spot, priority for ordering), `listing_candidates` (idempotent on `unique(source, sourceRef)` so re-polls just refresh metrics; carries `riskScore 0-100` + `riskFlags[]` + `status pending|listed|rejected|skipped` + linkage to `coins.id` / `web3_tokens.id` / `web3_networks.id`). Worker `api-server/src/lib/listing-discovery.ts` runs leader-gated every 15min: pulls from each enabled source (12s timeout, gentle 80ms throttle on DexScreener detail fetches), normalizes externals through a `sanitizeStr/sanitizeUrl` helper (strips control chars, hard-caps lengths so a malicious profile can't bloat rows), scores risk (very_new<3d=+25, new<14d=+10, low_liquidity<25K=+20, micro_cap<500K=+15, vol/mcap>5 anomaly=+15, |change|>50%=+10), applies the highest-priority matching active rule. Auto-list paths: `autoListSpot` does an `INSERT … ON CONFLICT (symbol) DO UPDATE … RETURNING id` upsert into `coins` (binance source so the existing price feed picks it up) — race-proof against parallel workers — and optionally creates a `pairs` row (`tradingEnabled=true`) vs `quoteSymbol`; `autoListWeb3` resolves the chain via a `chainKeyMap` (eth/bsc/polygon/arbitrum/optimism/base/avalanche/solana) and inserts into `web3_tokens` keyed on `priceCoinSymbol` (defaults to USDT). Auto-list failures fall back to `status='pending'` for manual review (try/catch around the entire auto-list block) so the candidate row is always recorded. Admin routes (`routes/listings-admin.ts`, all `ADMIN_GUARD`-gated): CRUD for rules + sources, paginated `/candidates?status=&chain=&search=` with stats grouping, `POST /candidates/:id/approve` (target=spot|web3, returns 409 "already listed" if status≠pending — prevents double-listing), `POST /candidates/:id/reject` with note, bulk reject/skip, and `POST /discover/run` for manual refresh. Numeric inputs validated (`Number.isFinite && >=0`) and string fields length-capped before DB insert; every mutation writes a `listing.*` audit log row. Public routes (`routes/listings.ts`, no auth, covered by global 600/min rate limit): `GET /listings/discover?limit&sort=volume|mcap|change|age&chain&search`, `GET /listings/trending` (top 20 by 24h change), `GET /listings/chains` (groups+counts). Admin UI at `/listings-admin` (Candidates / Rules / Sources tabs, Run-Discovery button, approve/reject inline). User UI at `/discover` (Hinglish copy "Naye coins aur trending tokens — multi-chain, risk flags dhyaan se padhna bhai!", 4 KPI cards, horizontal trending strip, filterable + sortable token table with risk pills, auto-refresh every 60s). 3 default rules seeded: CEX auto-list ($50M+ vol, scope=spot), Web3 auto-list ($500K+ vol + $250K+ liq, scope=web3), Manual fallback (anything else → pending).
- **User API Keys (HMAC-signed REST):** End-users can mint personal `zbx_…` keys to drive their account programmatically; the public scheme matches the docs page (`X-ZBX-APIKEY` / `X-ZBX-TIMESTAMP` / `X-ZBX-SIGN`, payload = `timestamp + METHOD + path + body`). Storage: `lib/db/src/schema/user-api-keys.ts` — `keyId` (24 hex public id), `secretEncrypted` (AES-256-GCM ciphertext of the 36-byte base64url secret; HMAC needs the plaintext back so bcrypt was wrong), `secretPreview` (last 4 chars for the listing UI), `permissions` jsonb (`read|trade|withdraw`), `ipWhitelist` jsonb, `status` (`active|disabled|revoked`), `expiresAt`, `lastUsedAt/Ip`. Crypto helpers (`api-server/src/lib/api-key-crypto.ts`) load a master key from `API_KEY_MASTER_SECRET` env or auto-generate one to `.local/api-key-master.key` (gitignored) on first boot, and use `crypto.timingSafeEqual` for signature compare. The HMAC middleware (`middlewares/api-key-auth.ts`) enforces ±5 min anti-replay, status/expiry/IP/perm gates, decrypts the secret, and verifies the signature against the raw body captured by an `express.json({ verify })` callback in `app.ts`. CRUD lives at `routes/account-api-keys.ts` (session-cookie auth, max 30 keys/user, refuses `withdraw` permission until 2FA is enabled, secret returned ONCE on create), and sample HMAC routes at `routes/v1.ts` (`GET /v1/system/time`, `GET /v1/account/me`, `GET /v1/account/balances` — joins `wallets` with `coins` to surface the symbol). Every mutation writes an audit log row (`apikey.{create,update,disable,enable,delete}`). UI: `artifacts/user-portal/src/components/settings/ApiKeysTab.tsx` is a 5th tab on Settings with the standard list / create-dialog (perm switches + comma-separated IP allow-list + optional expiry-in-days) / one-time secret reveal (copy buttons + ack checkbox) / disable / enable / delete (AlertDialog confirms).
